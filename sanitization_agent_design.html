Sanitization Agent — detect, pause for human review, then mask

The first real agent, redesigned around a human-in-the-loop pause. It chunks a document, uses the LLM to find what to mask and to draft the metadata the Tagging agent will later use, then stops and shows the whole proposal to a reviewer. Only on approval does it deterministically apply the masks and feed every confirmed entity back into the dictionary — so the next document leans on cheap deterministic matching first.

01

The big idea — two phases with a human gate between them

The single most important change from v1: the review is no longer a task filed at the end. The run splits in two and the agent physically stops in the middle until a reviewer approves. Detection is the LLM's job; application is deterministic code's job; the human sits exactly on the seam.

Phase 1 · propose

Detect & draft

  • Chunk the document by structural boundary.
  • LLM finds candidate entities — what to mask, its type, the exact location (chunk + character span), and a confidence.
  • LLM simultaneously drafts a sanitized summary + metadata for the Tagging agent to consume later.
  • Detects which client the document is about.
  • Run enters awaiting_review and stops. Nothing has been masked yet.
Phase 2 · apply (only on approval)

Mask & learn

  • Reviewer approves / edits / rejects the proposal in the UI.
  • On approve, a deterministic tool applies every confirmed span replacement — no LLM re-reading, so nothing is missed or altered.
  • Confirmed entities are written into the masking dictionary as approved and linked to the client.
  • Real identity is captured to the protected source registry; masked output + metadata are written; run completed.

Why detection and application are split across the human

Letting the LLM both decide and perform the masking means a low-confidence or hallucinated span could get masked (or a real one missed) with nobody checking. By making detection a proposal and application a deterministic replacement over reviewer-approved spans, the human sees the full picture once, and the actual text surgery is exact and repeatable — not a second fallible LLM pass.

02

Cold start — LLM-heavy now, deterministic-cheap later

Right now the dictionary is empty, so there is nothing to match deterministically — the first documents are almost entirely LLM work. That is expected and self-correcting.

A
Deterministic pass against approved dictionary entries. At cold start this finds nothing; every entity is a "gap."
B
LLM pass handles all the gaps — at first, that's the whole document.
C
Human approves. Every confirmed entity + its aliases is written to the dictionary as approved, linked to a client.
D
Next document: the deterministic pass now recognises those clients instantly and for free. The LLM only works the genuinely new residual. Cost per document falls as the dictionary grows.

The feedback loop is the whole point

The dictionary isn't seeded up front — it is grown by approved runs. Each review makes the next run cheaper and more consistent, and guarantees "the same client is always masked the same way" because the mask token is reused from the dictionary, never re-invented.

04

Chunking — how the document is fed to the LLM

You asked for chunk-wise processing. Here's exactly how, and why it stays precise enough to apply masks deterministically afterwards.

  • Chunk by structural boundary — page (PDF), slide (PPTX), or a small group of paragraphs (DOCX) — never arbitrary token windows. This is what lets progress read "page 47 / 100" and what keeps a location meaningful.
  • Record exact character spans. Every candidate carries (chunk_id, start_offset, end_offset, surface_text). That precise coordinate is what makes Phase 2's masking a deterministic string replacement rather than a guess.
  • Small overlap between chunks so a client name straddling a page break isn't lost at the seam.
  • Running "known entities so far" context. Each chunk's LLM call is given the compact list of what earlier chunks (and the dictionary) already found — so "Acme" in chunk 50 resolves to the same entity as "Acme Corp" in chunk 1, without stuffing 100 pages into one context window.
05

Sub-agents — a free pre-pass plus three specialist LLM passes

You asked whether this needs sub-agents. Yes — internally. The catalogue still shows one "Sanitization" agent; under the hood a free deterministic pre-pass feeds three role-specialised LLM passes. They are internal steps, not separately catalogued agents (that separation is the Coordinator's concern later).

Pre-pass · free, on-CPU

NER / regex

Before any LLM call: regex catches structured identifiers (emails, domains, IDs) exhaustively; a local NER model proposes org/person candidates. Zero tokens. Its output is deduped to the distinct candidate strings the Detector then classifies — this is the main cost lever (§10).

Sub-agent 1 · classify

Detector

Classifies the deduped candidate set (~50–100 items, not the whole document) with type + confidence, using the sanitization-detection skill. Still scans entity-dense chunks for anything the pre-pass missed. Flags low-confidence spans instead of guessing.

Sub-agent 2 · reduce

Reconciler

One pass over all chunk results. Merges duplicates, resolves aliases to one canonical entity, assigns a single consistent mask token per entity. Mostly deterministic (normalise + group); an LLM tie-breaker only for genuinely ambiguous cases.

Sub-agent 3 · document-level

Summarizer

Produces the sanitized summary + metadata for Tagging, and identifies the client. Runs over content with masks already substituted, so no raw identity leaks into the taggable metadata.

Why fan-out / reduce and not one giant prompt

A 100-page document won't fit one high-quality context, and detection must be exhaustive — every occurrence, not a best-effort skim. Splitting detection per chunk keeps each call sharp and lets them run in parallel; the Reconciler then guarantees cross-chunk consistency in a single deterministic sweep.

07

Metadata for Tagging — and the client-tagging guardrail

You want the run to produce metadata the Tagging agent uses, and you want to be able to "tag by client." Those two goals pull in opposite directions unless we split them carefully.

The tension — flagged, not silently resolved

Catalogue §04 guardrail: Tagging and general Search must never see client identity — that's the whole "show me Client X's documents" protection. If the client name goes into the taggable metadata, that guardrail breaks and any Search user could reconstruct client work by name.

Goes to Tagging

Sanitized metadata

Topic, domain, use-case, technology, geography, engagement type, and a masked summary. No client identity. This is the free-text/structured hint the Tagging agent classifies against its controlled vocabulary.

Stays protected

Client linkage

The run/document carries a protected client_account_id (via the source registry). "Tag by client" happens through this protected path — practice-leads/KM-governance filter by it; general Search cannot, because it never sees this field.

Net result — you get what you asked for, guardrail intact

You can still retrieve and organise documents by client, through the same protected client-lookup path §04 already describes. What you can't do is leak client identity into the general, un-permissioned Tagging/Search surface — which is exactly the protection the platform is built to guarantee.

Masking is code, detection is model

document.apply_masks is deliberately a protected, deterministic function — not an MCP tool. The model proposes spans; the code performs the exact replacement over the reviewer-approved set. That's how masking stays exhaustive and repeatable and how a stray model decision can never quietly write to the dictionary or alter the masked text.

Cost & token economics — and how we drive it down

Cost is essentially all LLM tokens (local storage, Postgres, compute are negligible). Model: Claude Sonnet 4.5 on Bedrock — $3 / 1M input, $15 / 1M output. A 100-page text document (~70K tokens of content) is the reference case.

Naive cost (no optimisation, cold start)

PassInput tokOutput tokCost
Deterministic dictionary pass0 (no LLM)0$0.00
Detector (20 chunks × ~5,650 in / ~600 out)113,00012,000$0.52
Reconciler (mostly code; occasional tie-break)5,0001,000$0.03
Summarizer (metadata, map-reduce)8,0002,000$0.05
Total~126,000~15,000≈ $0.60

Range $0.60–$1.00 depending on chunk size; ~$0.75 central. A 100-slide deck (~$0.25) is cheaper; a dense legal PDF (~$1.30) is more.

The one thing that can blow this up

If detection runs as a freely-looping tool-use conversation over one accumulating context, tokens grow quadratically (each turn re-sends the whole history) and cost can be 3–10× higher. Every estimate here assumes bounded per-chunk context — the single biggest cost lever, and the reason we chunk rather than hand the model the whole document.

The reduction levers (baked into the plan)

Structural win

NER/regex pre-pass → dedupe → classify unique set

A local regex/NER pass (free, on-CPU) catches structured identifiers — emails, domains, phone/account IDs — with zero LLM tokens, and generates org/person candidates. Dedupe to the distinct candidate strings (a client named 200× is one entity). The LLM then classifies only the unique set (~50–100 items), not the whole document. Cost becomes proportional to distinct entities, not page count — roughly flat as documents grow.

Free, no recall risk

Bedrock Batch + prompt caching + compact output

Batch inference −50% on all tokens — Sanitization is already async with a human pause, so zero latency cost. Prompt-cache the skill/system prompt + "known-entities" context (identical across chunks and docs; reads at 0.1× input). Compact output — output is 5× input cost/token, so return minimal JSON and never echo document text back.

Model tiering — held until measured

Haiku 4.5 ($1/$5, ~3× cheaper than Sonnet) could run the bulk detection scan, escalating to Sonnet only for ambiguous alias tie-breaks. Deferred until we've measured Haiku's detection recall on real docs — a missed entity is a leak, so this trades recall for cost and must be validated first.

Configuration≈ Cost / 100-pg doc
Naive (cold start, no optimisation)~$0.75
+ Batch (−50%) + prompt caching + compact output~$0.30
+ NER pre-pass + dedupe (classify unique set only)~$0.10–0.15

Honest note on "warm state"

As the dictionary fills, known clients mask for free — but the LLM still must read every page to catch new identifiers, so first-time-document API savings from the dictionary alone are modest (~10–25%). The dictionary's bigger payoff is human-review time and guaranteed consistency. The dedupe + NER + batch + caching levers above are what actually move the API bill.

11

Run lifecycle — the states, including the pause

The run is no longer a straight line to completed. It parks at awaiting_review until a human acts.

pending→ detecting→ awaiting_review→ applying→ completed
awaiting_review→ rejected · any state → failed
  • detecting — worker runs Detector fan-out → Reconciler → Summarizer, writing step rows as chunks finish.
  • awaiting_review — proposal (entities + locations + masks + summary + detected client) is filed; the worker releases the run and moves on. No thread is held.
  • applying — triggered by the reviewer's approval call; document.apply_masks runs, dictionary + registry are written. Cheap and deterministic, so it can run inline on approval.
  • rejected — reviewer declines; nothing is masked, no dictionary writes, run closed with the reason recorded.

Who reviews — not necessarily who submitted

Per the RBAC matrix, the mask/approval step is a KM-reviewer / KM-governance capability, and the identity that produced a run is never the one that approves it. A delivery user can submit a document, but the approval that promotes masks to approved and applies them is a reviewer's action.

14

Execution model

  • DB-polling worker, no broker. POST /runs inserts pending and returns; a worker loop executes detection. No Redis/Celery at this scale.
  • The pause needs no held thread. The worker files the review and releases the run — awaiting_review is just a row state. Approval is a separate inbound request that drives Phase 2.
  • Incremental step rows written as each chunk finishes, so GET /runs/{id} is a live progress view on long documents.
  • Application is inline & deterministic — cheap enough to run inside the approval request; no second background hop needed.
