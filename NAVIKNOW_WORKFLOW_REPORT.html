1System architecture

NaviKnow is deliberately simple infrastructure carrying deliberately careful agents. Four processes, one database, one LLM provider:

ComponentWhat it isRole
API serveruvicorn app.main:app (FastAPI, port 8000)Auth (JWT + roles), uploads, run creation, review decisions, remediation, governance endpoints. Reviewer-approved apply() phases execute here, synchronously in the request.
Background workerpython -m app.workerPolls Postgres every 2s for pending runs and executes detect(). Also runs the stale-run reaper (fails any run stuck at a transient status >15 min, with a 60s heartbeat so slow-but-alive runs are never reaped). Both processes must be restarted together after backend changes.
Postgreslocalhost:5433 (Docker)The only queue and the only state store — runs, steps, flags, review items, masking dictionary, logo references, tag vocabulary, audit log. No Redis/Celery, appropriate at current scale.
AWS BedrockConverse API, model from BEDROCK_MODEL_IDCurrently global.anthropic.claude-haiku-4-5 for everything — text detection, vision/OCR, summarization, tag classification. There is no separate vision model.
FrontendReact + Vite (port 5173)Sidebar app shell; Documents, Runs (with verification panel), Review queue/detail, Tags governance, Admin (users, accounts, masking dictionary).

A design decision worth recording: the platform originally exposed agent tools through an MCP server (naviknow-mcp, stdio subprocess + JSON-RPC). That layer was removed — tools are now plain in-process Python functions passed directly into the Bedrock tool-use loop. Same guardrail (protected calls are simply never in the model's dispatch table), zero subprocess overhead, one less dependency.

2The end-to-end pipeline

One click on "Run full pipeline" drives a document through the whole flow. Purple = a human decides; green = automatic; red = hard stop.

Upload document→ Coordinator starts Sanitization
(with auto-chain marker)→ detect(): NER → dictionary sweep → LLM → vision scan→ Human review: entities, images, style, client→ apply(): mask → render → redact → scrub ×3 → verify ×5
↳ All 5 channels clean → status completed → Tagging auto-starts or Anything survives → completed_with_issues → Tagging withheld→ One-click remediation fixes flagged content in place, re-verifies — if clean, auto-chain fires
↳ Tagging classifies the sanitized summary against the controlled vocabulary→ Review low-confidence tags & new-term proposals→ Tags applied — document is searchable

The core safety property of the whole pipeline: Tagging can only ever receive a Sanitization run whose status is exactly completed — which by construction means post-render verification passed on all five channels with zero blocking flags. "Analyse whether the approved selections were properly masked, then proceed" is not a policy anyone follows; it is the only path that exists. This is enforced twice: the Tagging agent itself refuses any source run that isn't completed, and the Coordinator's auto-chain hook only fires on completed.

3A-01 Sanitization — deep dive

Purpose

Remove or replace every client-identifying string and image in a document — consistently (the same client gets the same global mask token in every document, forever) — and prove it did so by re-checking its own output. Two-phase: detect() proposes, a human reviews, apply() commits deterministically.

Phase 1 — detect() (runs on the worker)

  1. Extract + NER pre-pass (~2s, free) — the document is split into structural chunks (PDF pages, DOCX paragraph groups + headers/footers + text boxes + footnotes, PPTX slides with recursive group-shape traversal + speaker notes, XLSX sheets including tab names and print headers). Regex + Presidio surface candidate entity strings. A scanned PDF with no text layer raises a warning flag — the text channel is honestly blind there, and the reviewer is told so.
  2. Deterministic dictionary pass + full-text sweep (<0.1s, free) — every candidate is looked up in the global masking dictionary; then, independently of what any model surfaced, every approved entity's aliases are regex-swept against the entire document text. Once an entity is approved, its masking never again depends on per-run model recall. (This sweep exists because a weak LLM run once left 6 already-approved names unmasked in a "clean" file — the dictionary knew them; nothing had asked it.)
  3. LLM Detector (~8s, ~9k tokens) — a Bedrock tool-use loop where the model may call fs_read_document (a plain in-process function, the agent's only model-callable tool) to read chunk ranges, then returns client-identifying surface strings with type + confidence. Surfaces only — exact spans are found deterministically afterward, because LLMs are unreliable at offsets. Candidates below the 60% confidence threshold are auto-excluded with an info flag (visible, re-addable, never silently dropped).
  4. Image scan: vision + OCR + logo hash (~93s — the dominant cost; see §9) — every embedded image is enumerated via a raw zip glob (masters, layouts, orphaned media included), deduplicated by SHA-256 and then perceptually (Hamming ≤ 2 clusters near-identical renditions into one scan). Each cluster gets one Bedrock vision call (OCR text + client-identity judgment), one perceptual hash matched against the logo-reference library, and three deterministic overrides: a confirmed OCR/dictionary hit or a confident logo-hash match (distance ≤ 4) forces the flag on regardless of the model's own judgment; an own-firm hit (name or logo phash) forces it off; distances 5–10 or mid-band confidence mark needs_human_judgment. A confident match to an already-approved entity becomes mandatory_redaction — a governance decision the reviewer's checkbox cannot re-open.
  5. Summarize + file review (~7s) — a sanitized, identity-free summary and metadata hints (domain, use cases, technologies, geography, engagement type) are generated for Tagging, and a review item is filed. The run parks at awaiting_review. Detect never mutates the approved dictionary — new entities enter as pending_approval.

Phase 2 — apply() (runs on approval, in the API server)

  1. Consume the reviewer's decision — removed surfaces, reviewer-added entities, per-image include/exclude flips (except mandatory redactions), the masking style (token / black-out / remove), and the single explicit "this is the client" designation for account linkage. The raw edits payload is persisted for audit.
  2. Resolve global mask tokens — each approved surface gets (or reuses) its permanent [CLIENT_N] token; approval here is what admits an entity into the global dictionary and future full-text sweeps.
  3. Mask text + render in the original format — word-boundary-safe, longest-surface-first, whitespace-tolerant regex across DOCX runs/tables/headers/text boxes, PPTX shapes (recursive)/tables/notes, XLSX cells/sheet names/print headers, and PDF via a true word-level matcher (consecutive-word matching within a text block — handles names wrapped across lines, immune to substring false-positives). XLSX image redaction happens inside this same render pass because openpyxl renumbers media partnames on save.
  4. Redact approved images — media-part byte swap (docx/pptx/xlsx) or redaction rects (PDF), resolved by each cluster's full occurrence index list so every rendition of a merged near-duplicate is covered.
  5. Scrub the side channels — document properties (core/app/custom + PDF XMP), hyperlink targets (rewritten in the run's masking style, via direct _rels editing), and comments/track-changes/annotations (element-text-only substitution — never a full XML re-serialize, which can corrupt Word files).
  6. Verify — 5 independent channels (~62s, dominated by re-scanning images) — each channel re-derives its answer from the rendered file, sharing no state with the step it checks: text (re-extract + regex), images (full re-scan with placeholder recognition; structured residuals persisted for remediation), metadata, comments, hyperlinks. Any residual → a named blocking flag → completed_with_issues. All clean → completed, and the auto-chain fires if a Coordinator started this run. Approved image redactions also feed the logo-reference library, so future documents get deterministic logo matches.

The remediation path

A completed_with_issues run shows a one-click "Re-run sanitization on the flagged content" button. It fixes the already-rendered file in place, scoped to exactly the flagged slides/pages (zero new vision calls when structured residuals were persisted), re-scrubs the side channels, re-verifies everything, clears superseded flags, and — since it is a second path to completed — triggers the same auto-chain hook. The one honest limitation, stated in the UI: body-text residuals can't be fixed in place and need a full re-run.

Guardrails (structural, not prompt-based)

  • The model can call exactly one tool. The masking dictionary, review filing, source registry, and audit log are plain agent-code calls it cannot reach.
  • Own-firm names/logos are excluded by three independent signals (normalized text check, prompt injection, perceptual hash) — no single failure re-masks your own brand.
  • mandatory_redaction: a reviewer cannot un-redact a confirmed match to an already-approved entity — that governance decision was already made once.
  • Every run files a review item; there is no confidence level that skips the human.
  • Real client identity is captured into a protected registry before masking and never appears in output_json.

4A-02 Tagging — deep dive

Purpose

Classify a completed Sanitization run's sanitized output against the controlled tag vocabulary (domain, use_case, technology, geography, engagement_type) so documents become searchable by topic — never by client.

Process

  1. Load source — hard gate: the referenced Sanitization run must have status exactly completed. The agent only ever sees the sanitized summary + metadata hints; it has no path to the original document, the masking dictionary, or the source registry — it could not reconstruct client identity even if instructed to.
  2. Classify (LLM) (~3s, ~1.5k tokens, ~$0.004) — one Bedrock call: vocabulary + sanitized summary + hints in, tags with confidence out. The prompt instructs "prefer existing vocabulary verbatim; propose new lowercase terms only when nothing fits".
  3. Validate (deterministic hard gate) — every proposed tag is checked against the approved vocabulary in the database. In-vocabulary + confidence ≥ 0.7 → auto-applicable. In-vocabulary but below 0.7 → flagged for reviewer confirmation. Unknown term → recorded as a pending_approval vocabulary proposal, never applied to the document. The model physically cannot attach an out-of-vocabulary tag.
  4. Conditional review — if everything was confident and in-vocabulary, the run auto-applies with no human step; any flagged tag or new-term proposal parks it at awaiting_review where the reviewer can untick tags.
  5. Apply — surviving tags are written to run_tags; pending new terms stay in the governance queue on the Tags page.

The vocabulary loop

The vocabulary is currently governance-built from empty (startup seeding was deliberately disabled at your request). Practice-leads propose; KM-governance/admin approve, add directly, or reject on the Tags page. 35 proposed terms are sitting in that pending queue right now — approving the good ones is the single highest-leverage action available (§9, R3).

5A-04 Coordinator — deep dive

Purpose

No content work of its own — it turns two manual actions into one. Deliberately minimal against the full design-doc spec: no eligibility-rule engine yet, no separate pipeline table. Its entire mechanism is one marker plus one hook:

  1. Start Sanitization with the marker — the Coordinator creates a child Sanitization run whose input carries auto_chain_to: "tagging" and drives it through the same run_detection() the worker uses. Zero duplicated logic; every Sanitization guardrail applies unchanged. Its own run completes immediately (the UI hops straight to the child run).
  2. Pause-and-resume — if Sanitization needs review, it parks at awaiting_review like any other run. The reviewer uses the normal Review UI; nothing new to learn. The run page notes that Tagging will follow automatically.
  3. The auto-chain hook — when any Sanitization run finalizes at exactly completed and carries the marker, a Tagging run is enqueued (idempotent — a re-fire can never create a duplicate). The hook sits in the one function both completion paths share, and in the remediation path (a gap found and fixed during live testing: a document fixed via remediation rather than approved-clean-first-try would otherwise never have chained).

Verified live, end-to-end: Coordinator → Sanitization → paused for review → approved → landed completed_with_issues (3 flagged images) → chain correctly withheld → remediated → completed → Tagging auto-enqueued → the live worker picked it up unaided and produced a 15-tag proposal.

What it deliberately does not do yet

  • eligibility.check (deterministic contract-restriction gate) — no rules table or UI exists to configure it against; building it now would be speculation.
  • Orchestrating A-03/A-05/A-06 — those agents don't exist yet.

9Optimization research — prioritized

Ordered by measured impact ÷ effort. R1–R4 are code changes; R3 is a governance action only you can take.

R1 — Parallelize the vision scans (bounded concurrency)

P1 · ~4× faster runs

Both the detect-phase scan (92.9s) and the verification re-scan (62.2s) await one Bedrock vision call at a time inside a loop. The calls are independent per image cluster. Running them with asyncio.gather under a semaphore of ~6–8 concurrent calls would take the measured ~155s of vision time to roughly 20–30s — a 4.5-minute run becomes ~1.5 minutes, with zero cost change and zero quality change. Watch Bedrock throttling (429s) and add simple retry-with-backoff; the existing per-call error handling already degrades a failed scan to "flag for human review" rather than failing the run. This is the single biggest experience improvement available.

R2 — First-run recall insurance: double-pass detection union

P1 · closes the remaining leak surface

The measured 11-vs-3 detector variance is fully neutralized for known entities by the full-text dictionary sweep — but a document's first pass through the system has no dictionary history for its never-seen names. For documents where the sweep contributes zero known entities, run the LLM detector twice and union the proposals. Cost: ~$0.01 and ~8s (parallelizable with R1's machinery). The union can only add candidates for the reviewer to see — the human gate means over-proposal is cheap and under-proposal is a leak. Skip the second pass when the sweep already found entities (repeat documents), keeping steady-state cost unchanged.

R3 — Curate the 35 pending vocabulary terms (you, ~10 minutes)

P1 · governance, highest leverage per minute

The vocabulary was rebuilt from empty, so the classifier's recent runs proposed everything as new terms — 35 now sit at pending_approval on the Tags page (finance domain, a rich set of finance use-cases, generative/agentic AI technologies, global/multi-jurisdiction geographies). Until approved, Tagging applies nothing and every run needs review. Approve the good ones once and: tags start auto-applying at high confidence, review load drops, and search granularity jumps. Also worth merging near-duplicates while you're there (scenario modelling/scenario modeling, reconciliation/reconciliation automation, multi-jurisdiction/multi-jurisdictional) — duplicates fragment search.

R4 — Turn on prompt caching (it's already built)

P2 · ~30–40% detector input-cost cut

converse_with_tools supports a cachePoint after the stable system prompt (cached reads bill ~0.1×), but no caller passes cache_system_prompt=True. The detector's system prompt + schema (~1.5k tokens) is resent on every loop iteration and every document. One-line changes in the detector, summarizer, and classifier calls; savings compound with R2's second pass and grow with volume.

R5 — Clear detect-phase blocking flags once verification passes clean

P2 · trust/UX

Recurring on every clean run: the detect-time awareness flag ("N embedded images appear to reveal the client") stays visible after all N were redacted and verified, making verified-clean runs look unsafe to anyone reading the flag list. The remediation path already clears superseded flags — _finalize_completed should do the same when all five channels pass. Small change; disproportionate credibility gain in demos.

R6 — Honest tag-review semantics

P2 · Tagging quality

Three related fixes: (a) run-level approval currently blanket-marks every tag confirmed, including below-threshold ones the reviewer never individually touched — flagged tags should keep a distinguishable status through approval; (b) run confidence is min() over tags, so one weak tag mislabels a strong twelve-tag run — report mean + count-below-threshold instead; (c) "domain is single-valued" is stated in the prompt but unenforced (recent runs applied two domains, which was actually better for search) — make multi-domain the explicit policy or enforce single in code; either beats per-run model whim.

R7 — Output reporting: split text-masked vs image-derived entities

P3 · reporting clarity

"29 entities masked / 7 text occurrences" reads like over-reporting until you know 22 were image-only identities redacted as pixels. Add a source field (text / ocr / logo) to entities_masked so run outputs reconcile at a glance. Same spirit: the detector step label still says "detect (LLM + MCP)" — MCP is gone; relabel.

R8 — Summarizer policy: refer to the delivery firm generically

P3 · consistency

Summaries alternate between "Navikenz presents…" and "A global consulting firm presents…" run to run. Not a leak (own-firm is deliberately unmasked), but inconsistent and zero search signal when every document is your own. One sentence in the summarizer prompt fixes it.

R9 — Tune thresholds with accumulated evidence

P3 · needs data, not code

The 0.6 sanitization auto-exclude and 0.7 tag threshold are reasoned defaults, not measured ones. Once ~20 reviewed runs exist, compare confidences of entities/tags reviewers added back vs. ones nobody missed, and tune. Early signal so far favors the current cutoffs (auto-excluded candidates have gone unchallenged).

R10 — Roadmap: the remaining catalog agents, in dependency order

Next builds

A-05 Search/Retrieval is the natural next build: it needs only what already exists (sanitized summaries + tags in Postgres) — start with SQL/keyword + tag filtering over document_metadata + run_tags before reaching for embeddings, and inherit the structural guardrail for free (it simply never gets the dictionary or registry in reach; client-name-shaped queries decline gracefully). A-06 Deck Drafting waits for a slide-level index, which itself waits on real sanitized volume — correct to defer. A-03 SharePoint Cleanup is blocked only on a Microsoft Graph app registration; per its own design it should be built against a mocked graph.list_sites first, so it can be developed any time without access. The Coordinator's eligibility.check gate becomes worth building when the first real contract restriction needs encoding — as a rules table + deterministic check, never an LLM judgment.
