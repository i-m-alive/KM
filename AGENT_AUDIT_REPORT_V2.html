1What this audit did

Rather than re-checking the pass/fail channels (which all read green), this audit went after the question the channels can't answer: is the green actually trustworthy, and is the output quality consistent? Method:

  • Pulled the two newest real runs of each agent from the database and compared them against the immediately preceding pair — same document, 27 minutes apart, giving a natural A/B of run-to-run consistency.
  • Diffed the actual masked surfaces between the two Sanitization runs (from masking_occurrences), then swept the second run's rendered file for anything the first run masked but the second didn't.
  • Read the full Tagging agent code path (untouched in previous audits) against its observed database rows.
  • Inspected the masking dictionary's real contents (68 entities) for quality, not just count.

2The critical finding — a "clean" run that wasn't

Approved dictionary entities silently went unmasked when the LLM had a weak run

FIXED during this audit · severity: leak-class · found by diffing two runs of the same document

The observation: run 9bd3c579 masked 13 text occurrences on the Fidelity deck; run 72f44588 — same document, 27 minutes later — masked only 7, and both reported verified_text: true. Diffing the masked surfaces and sweeping run 2's rendered file found 6 company names present in the output that run 1 had masked: BlackRock, GSK, Siemens, Tata Capital, Arvind Fashions, VFS — plus client system names (Aladdin, SHERPA X) also missed. All of these are approved entities in the global masking dictionary.

The three-link failure chain:

  • Haiku detector variance: the LLM proposed 11 entities in run 1 but only 3 in run 2 — the same variance §7 of the previous report predicted, now measured on identical input.
  • The dictionary pass was hostage to the candidate list: it only ran dictionary.lookup() on strings the NER pre-pass happened to surface (8 candidates). If nothing re-proposed "BlackRock" this run, the dictionary — which knew BlackRock — was never asked.
  • Verification only checks proposed surfaces: verified_text re-checks that the surfaces in this run's proposal are gone. It structurally cannot catch what detection never proposed. The green channel was honest about what it checked, but what it checked was too narrow.

The fix (shipped and live): a full-text dictionary sweep in detect() — every approved entity's aliases are regex-scanned (word-boundary, case-insensitive) against the entire document text, and any hit joins the proposal deterministically at confidence 1.0, regardless of what NER or the LLM did that run. Once an entity is approved in the global dictionary, its masking never again depends on per-run model behavior. A second latent bug was fixed in the same pass: known entities were masked using aliases[0] rather than the alias actually matched in the document — an entity whose first alias is a different variant would have masked the wrong string.

Validation: the sweep, run offline against the original deck, finds 18 approved entities — including all 6 survivors and both client system names. The 43-check regression suite still passes; both servers restarted with the fix.

Residual risk (honest): the sweep protects anything ever approved. A first-time document with never-before-seen third-party names still gets only the LLM's variable recall on those names. See the double-pass recommendation in §5.

3All other findings

F2 — Junk OCR-split entities in the masking dictionary now mask generic English words

Governance action needed — deliberately NOT auto-fixed

The dictionary contains approved entities that are OCR fragments of real names: [CLIENT_44] "ARVIND", [CLIENT_45] "FASHIONS", [CLIENT_46] "TATA", [CLIENT_47] "CAPITAL", and [CLIENT_LOCATION_1] "Global Controlling Leadership". Run 2 actually masked the standalone word "Capital" — over-masking that corrupts innocuous text like "capital markets". Longest-first ordering means "Tata Capital" wins where the full phrase appears, but standalone occurrences of the bare words still get hit. The new full-text sweep makes these fire consistently on every future document, so cleanup is now urgent rather than cosmetic: open Admin → Masking dictionary and Skip the fragment entities (the skip mechanism built earlier exists for exactly this). Skipped entities are excluded from the sweep. This is a reviewer decision — which fragments are junk vs. genuinely maskable (e.g. "TATA" alone arguably is) — which is why it wasn't done unilaterally.

F3 — Run-level approval blanket-confirms flagged tags

Tagging · semantics bug · app/agents/tagging/agent.py

The latest Tagging run flagged use_case=fraud-detection at 0.65 (below the 0.7 threshold) for reviewer confirmation — but its database row reads status=confirmed. Cause: on approval, apply() sets every surviving tag to "confirmed", including ones the reviewer never individually looked at. The per-tag "flagged" state exists only until approval, then is erased. "Confirmed" should mean a human confirmed that tag; flagged tags not explicitly acted on should stay distinguishable (e.g. keep flagged, or a confirmed_bulk status), or the review UI should require per-tag confirmation for below-threshold tags.

F4 — "domain is single-valued" is stated in the prompt but not enforced anywhere

Tagging · policy drift

The classifier prompt says domain and engagement_type are single-valued; the latest run applied two domain tags (financial-services 0.95 + asset-management 0.85) and nothing objected. Here the result was arguably better for search — which is exactly why this needs a deliberate policy decision rather than silent drift: either allow multi-domain (update the prompt and the summarizer's matching instruction), or enforce single-valued in code (keep the highest-confidence one). Currently the model decides per-run, which is the worst option for consistency.

F5 — Detect-phase blocking flag lingers on verified-clean runs (recurring)

Sanitization · UX/trust · now observed on 3 consecutive completed runs

"30 embedded image(s) appear to reveal the client — review before treating this document as sanitized" still shows on a run where all 30 were subsequently redacted and all five channels verified clean. Anyone reading the flags list concludes the file is unsafe when it isn't. The remediation path already clears superseded flags; the normal apply path should do the same when verification passes clean.

F6 — use_case vocabulary is still too coarse for the documents actually flowing through

Tagging · vocabulary coverage

Sanitization's metadata hints listed 10 concrete finance use-cases (record-to-report automation, regulatory reporting, reconciliation automation, consolidation and close, cash application matching, …). The classifier could only map them to automation + analytics + fraud-detection — the searchable granularity is being flattened at the last step. Meanwhile technology=generative-ai and technology=ai sit at pending_approval in the existing Tag Vocabulary page's queue with nobody having approved them (which is also why this run's pending_new_terms is empty — the term exists, it's just stuck). Approve/curate the queue and seed the use_case category with the recurring finance-transformation terms.

F7 — Output reporting conflates text-masked and image-only entities

Sanitization · reporting clarity

The run output says "29 entities masked" but only 7 text occurrences across 6 entities — the other 23 are image/OCR-derived identities that were redacted as pixels, not text. Nothing distinguishes them in entities_masked, which makes every run look like it over-reports. Split the output into text_entities / image_entities (or add a source field) so the numbers reconcile at a glance.

F8 — The sanitized summary names the own firm, inconsistently

Sanitization · policy decision

Run 1's summary began "A global consulting firm presents…"; run 2's begins "Navikenz presents…". Not a client-identity leak (own-firm is deliberately unmasked), but it's run-to-run inconsistent and adds zero search signal when every document is the firm's own. Pick a policy — probably "refer to the delivery firm generically in summaries" — and put it in the summarizer prompt.

F9 — Run confidence = the single weakest tag

Tagging · metric

The Tagging run's confidence is min() over all tags, so one 0.65 tag among twelve strong ones labels the whole run 0.65. Twelve-tag runs will almost always contain one weak tag, so the run-level number trends meaninglessly low. A mean (or the count of below-threshold tags) would describe the run honestly.

4What's confirmed working — validated this audit, not assumed

  • All five verification channels green on both Sanitization runs, and the raw-XML sweep of every package part confirms zero occurrences of the client name in both rendered files.
  • Hyperlink-target scrubbing fired in production for the first time — step 8 of run 72f44588: "1 hyperlink target(s) rewritten", and the hyperlinks channel verified clean. The feature built last round works on real documents, not just fixtures.
  • Perceptual dedup + index-based redaction: 50 unique clusters scanned (not 57+ raw images), 39 images blacked out, no survivors of the resolution bug fixed last round.
  • The vocabulary expansion keeps paying off: domain=financial-services (0.95), asset-management (0.85), geography=global (0.88) all applied — the exact terms that were unavailable two audits ago.
  • Own-firm exclusion held: Navikenz appears in the summary (see F8) but was never proposed as a maskable client entity in either run.
  • Cost discipline: ~$0.057 Sanitization + ~$0.003 Tagging per full pipeline on Haiku.
