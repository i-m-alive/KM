1How the pipeline works

Sanitization is a two-phase background agent: detect() proposes what to mask and pauses for human review; apply() deterministically masks, renders, and re-verifies on approval.

extract text→ NER pre-pass (regex+Presidio)→ dictionary lookup→ LLM Detector (MCP)→ scan images (vision+OCR+phash)→ human review
apply masks (regex, word-boundary)→ render native format→ redact images→ scrub metadata→ verify (5 channels)

The five independent verification channels re-derive their answer from the rendered file rather than trusting the render step succeeded — this is what turns "did masking work" from a hope into a checked fact:

ChannelWhat it re-checksAuto-fix on failure?
verified_textMasked surfaces don't survive in extracted textYes — full masking pipeline
verified_imagesRe-scans every image in the rendered file for logos/wordmarksYes — image redaction + mandatory_redaction
verified_metadataCore/app/custom docProps propertiesYes — metadata scrubbing
verified_commentsComments, track-changes deletions, PDF annotationsNo detect-and-block only
verified_hyperlinksHyperlink target URLs (not just display text)No detect-and-block only

3Everything that went wrong (and why)

In the order they were actually found, working end-to-end against the real test document. Each was reproduced and fixed, not just theorized.

1. Grouped shapes in PPTX were invisible to the whole pipeline

Fixed · Root cause

What happened: A plain for shape in slide.shapes loop never descends into PowerPoint "Group" shapes — a very common pattern for logo+label compositions. Extraction, masking, and image-scanning all silently skipped anything nested inside a group.

Fix: pptx_walk.py's iter_shapes_recursive(), wired into extract/render/image-scan uniformly so all three modules agree on what a "shape" is.

2. Image enumeration missed slide masters, layouts, headers/footers, thumbnails

Fixed · Root cause of the original "GSK logo not masked" report

What happened: Shape-tree walking only sees what's directly on a slide. A logo baked into a slide master (reused across every slide) or the document's auto-generated thumbnail was never enumerated at all.

Fix: A raw zip glob over */media/* (plus docProps/thumbnail.*) as a second enumeration layer — proven more complete than relationship-graph walking, since an orphaned media part with no live relationship still physically sits in the zip.

3. Client-account linkage auto-guessed from "whichever entity appeared first"

Fixed · Found via direct DB inspection (11 unrelated entities all linked to one account)

Fix: Client-account linkage now requires an explicit, single reviewer-designated client_entity_surface — no more inference.

4. Masking regex had no word boundaries — corrupted ordinary English

Fixed · Most damaging bug found this session

What happened: A short masked surface like "RIA" or "sure" matched as a bare substring, mangling unrelated words throughout the document: "Va[RIA]nce" (Variance), "expo[sure]" (exposure), "clo[sure]" (closure), "t[ria]l balance" (trial balance). This wasn't a privacy bug — it was silently corrupting the delivered document's prose.

Fix: A shared surface_pattern() helper (\b...\b) applied consistently across masking, verification, metadata scrubbing, and the PDF redaction path — the PDF path uses a padded-context check since PyMuPDF's search_for has no native word-boundary support.

5. The delivery firm's own name/logo got masked as if it were the client

Fixed · Multi-layered, needed 3 separate fixes

What happened: "NAVIKENZ" (the presenting firm's own name, used in a "who we are" self-intro slide) and its own wordmark logo ("NAVICADE") got detected and masked as client identity. Worse: once wrongly approved, they became permanent global dictionary entries, re-corrupting every future document.

Fix — three layers, because one wasn't enough:

  • is_own_firm() — a configurable, normalized "contains this token" check (not a bare prefix — a prefix match on "navi" would also wrongly exclude real unrelated companies like Navistar/Navient), applied at every entity-merge point.
  • Injected directly into the vision-scan prompt so the model's own judgment knows the specific name, not just a generic "don't flag our own branding" instruction.
  • A hard deterministic override — the vision model's contains_client_identity boolean was observed disagreeing with its own description text ("this does NOT reveal client identity" while still flagging true). Prompt instructions alone aren't reliable enough; the override forces the flag false regardless of the model's own wording.

6. OCR text extraction is unreliable — same logo, different results call to call

Fixed

What happened: A stylized multi-color "NAVICADE" wordmark was correctly flagged contains_client_identity=true at 99% confidence — but with empty ocr_text. The text-based own-firm override had nothing to match against.

Fix: A second, OCR-independent signal — perceptual-hash matching against known own-firm logo images, reusing the same phash mechanism already built for client-logo recognition, just inverted for exclusion.

7. A confirmed logo match kept getting excluded by the reviewer — three runs in a row

Fixed · Root cause: misleading model commentary, not reviewer error

What happened: "NVISION" (a genuine maskable entity, confirmed via exact perceptual-hash match, distance 0) kept getting unchecked during review — because the vision model's own description text persistently argued "this appears to be the delivery firm's own branding... does not reveal client identity", even after the description was rewritten to lead with a "CONFIDENT MATCH" headline.

Fix: Stopped relying on the reviewer reading AI-generated commentary correctly. A confirmed phash match (distance ≤ threshold) to an already-approved dictionary entity is now a mandatory_redaction — the checkbox is visually locked in the UI, and apply() ignores excluded_image_groups for these at the backend level regardless of what the review request contains. This is enforced server-side, not just hidden in the UI.

8. Recurring low-confidence noise on every single run

Fixed

What happened: Short OCR fragments and industry acronyms ("RIA", "TMG", "GMR", "gsk") kept resurfacing as separate low-confidence warnings on every run of the same document.

Fix: A configurable SANITIZATION_CONFIDENCE_THRESHOLD (default 0.6) — candidates below it are excluded from the proposal entirely (one aggregated info line instead of N warnings), reviewer can still add any back manually.

9. Metadata leaks (document properties never touched by text masking)

Fixed · Real leaks found: "BAJAJ" in core properties, "RIA" in Company/Manager

Fix: A dedicated metadata-scrub step rewrites docProps/core.xml, app.xml, and custom.xml (the same three parts the verifier reads) using the same masking style already chosen for the run.

10. Image scan cap silently dropped most of a real deck

Fixed

What happened: MAX_IMAGES_SCANNED = 30 meant 27 of this deck's 57 images were never even scanned — not "unlikely to be a logo," genuinely unverified.

Fix: Raised to 150 for realistic deck sizes.

11. numpy int64 poisoned JSON serialization mid-run

Fixed · Manifested as an opaque "unhandled errors in a TaskGroup" failure

What happened: ImageHash.__sub__ (the Hamming-distance subtraction for logo matching) returns numpy.int64, not a plain int — looks and prints like an int, but isn't JSON-serializable. The moment a logo match fired, saving the proposal to the JSONB column crashed the entire run.

Compounding bug: the failure-handling path itself then crashed again (tried to commit on an already-rolled-back session), so the run was never even marked "failed" — it sat at "detecting" forever with zero record of what went wrong.

Fix: Explicit int() cast at the source; mark_failed() now rolls back before committing, so a crash can never again leave a run silently stuck.

12. Two independent backend processes, only one gets restarted

Process fix, not code · Cost real debugging time, twice

What happened: detect() only ever runs inside a separate app.worker polling process; apply() runs inside the uvicorn request handler. Restarting only one after a code change meant detect() kept running stale code (reporting "0 images found") while apply()'s verification — on fresh code — correctly found leaks the stale detector never had a chance to flag.

Fix: Both processes now get restarted together after every backend change, as standard practice for this project.

13. Killing the worker mid-run orphans that run permanently

Known limitation

No stale-run recovery mechanism exists. A run interrupted mid-detect() sits at a non-terminal status forever — indistinguishable from "still working" without manually checking process CPU time. See recommendation in §6.

14. Bedrock model deprecation and malformed structured output

Fixed · Model-dependent, not a NaviKnow bug per se

What happened: The original Sonnet model ID was retired by AWS Bedrock mid-project ("reached end of its life"). Separately, when testing Haiku 4.5 for cost, it periodically echoed the JSON schema as one complete object then appended the real answer as a second, comma-glued object — technically invalid JSON.

Fix: Verified current model IDs directly against the account's list_inference_profiles rather than guessing; added a targeted JSON-repair heuristic for the specific schema-echo failure shape.

15. Hyperlink target leaks (detect-and-block, no fix path)

Open — deliberate scope decision

What happened: "NVISION" — a genuine maskable term — appeared as a hyperlink target (the href, not the visible text). Hyperlink scrubbing was explicitly scoped out (detect-and-block only) earlier in the project, so this channel can block a run indefinitely with no way to reach "clean."

Status: Offered to build hyperlink-target scrubbing (rewrite the href the same way text is masked); not yet actioned. See §5.

4Current state — what's solid vs. fragile

Solid

  • Text masking across DOCX/PPTX/XLSX/PDF, word-boundary safe, format-agnostic.
  • Image enumeration (shape-tree + zip-glob) — genuinely complete, proven against the real deck's 57 images including masters/layouts/thumbnails.
  • Own-firm exclusion — three independent, layered signals (text, prompt injection, phash) so no single failure mode reintroduces the bug.
  • Multi-channel verification — a channel this doesn't check is a channel that can silently leak; five channels now re-derive from the rendered file every time.
  • mandatory_redaction enforcement — the one case where reviewer discretion is deliberately overridden by a governance decision already made (an approved dictionary entity).

Fragile

  • Vision-model non-determinism. The same image, scanned twice (once at detect time, once at verify time), can get different judgments. This is inherent to LLM vision calls, not a code bug — the verification system exists specifically to catch this, but it means occasional re-runs are expected, not a sign of something broken.
  • Reviewer trust in AI-generated commentary. Free-text descriptions have repeatedly proven unreliable enough to mislead a careful reviewer. Anywhere a deterministic signal exists, it should be load-bearing — description text should be supplementary only.
  • Haiku's structured-output reliability is measurably worse than Sonnet's for this task (empty fields, schema echoing, boolean/text inconsistency). The guardrails built this session compensate for it but don't eliminate the underlying gap.

6Optimization recommendations

Expand the tag controlled vocabulary

P1

Add financial-services (and/or asset-management) to the domain category, and global to geography — both confirmed missing by live validation, both likely to recur given the kind of documents actually being processed. This is a one-line governance action (via the existing tag vocabulary admin UI) with immediate, measurable effect on tag accuracy — the cheapest fix in this entire report relative to its impact.

Build hyperlink-target scrubbing

P1

Mirror the metadata-scrub approach: rewrite the actual href for any masked surface, using the same masking style already selected for the run, then flip verified_hyperlinks from detect-and-block to detect-and-fix. This closes the only channel that has actually caused repeat, unresolvable blocks in real usage.

Add a stale-run reaper to the worker

P2

A background check that requeues (or fails cleanly) any run stuck at a non-terminal status (detecting/working) past a reasonable timeout (e.g. 10 minutes). Removes the need to manually diagnose "is this actually stuck" via process CPU time.

Cut vision-scan cost via perceptual, not just exact, deduplication

P2

Images are currently deduped by exact SHA-256 before scanning. Near-identical images (same logo, different compression/resize) are treated as separate Bedrock calls. Grouping by perceptual hash first (with a tight threshold) before falling back to exact-hash for the render-time redaction match would reduce the number of vision calls on decks with repeated, slightly-varying logo instances — directly reduces both cost and the scan-to-scan non-determinism surface area.

Build a fixed regression corpus

P2

Every bug this session was found by re-running the same real document repeatedly. Formalizing 3-5 representative documents (one with grouped shapes, one with masters/layouts, one with a scanned/no-text-layer PDF, one with hyperlinks/comments) as a standing regression suite would catch these classes of bugs before they reach a real run, rather than after.

Periodic dictionary hygiene

P3

The masking dictionary view (built this session) now makes cleanup possible, but nothing prompts it. A lightweight periodic review of pending_approval entries older than N days — auto-flagging candidates for governance attention — would prevent the same "REDACTED gray box" -class of forgotten-decision entities from silently accumulating.

Tune the confidence threshold with real data

P3

0.6 was a reasonable starting default, not a measured one. Once a handful of real documents have gone through the pipeline, look at the distribution of confidences on entities that reviewers subsequently added back manually (should be rare) vs. ones nobody added back (validates the cutoff) to tune it with evidence.

7Model choice: Sonnet vs. Haiku

SonnetHaiku
Cost (this session's real run)~$0.24 per full run (57 images)~5x cheaper per token
Structured-output reliabilityConsistently well-formed JSONOccasional schema-echo malformation (auto-repaired)
Vision field completenessReliably fills description/OCR fieldsSometimes empty despite correct boolean judgment
Boolean/text self-consistencyMore consistentObserved disagreeing with its own description text more often
Net effect with current guardrailsFewer edge cases hit at allSame eventual correctness, more reliance on deterministic overrides doing the real work

The deterministic guardrails built this session (own-firm phash, mandatory_redaction, confidence-threshold auto-skip) were specifically designed so correctness doesn't depend on the model's own judgment being reliable — this is what makes Haiku viable for cost-sensitive runs at all. Without them, Haiku's reliability gaps would show up as real, undetected leaks rather than caught-and-corrected ones.
