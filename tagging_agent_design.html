NaviKnow · Agent Platform A-02 Detailed Design

Tagging Agent — classify sanitized content against a fixed, governed vocabulary

The second real agent, built alongside Sanitization. It never sees a raw document — only Sanitization's masked output — and maps that content to a controlled tag taxonomy (domain, use-case, technology, geography, engagement type). Client name is deliberately not a tag category, and a deterministic validator makes an out-of-vocabulary tag structurally impossible to emit.

01

Where it sits — downstream of Sanitization

Tagging consumes what Sanitization produces. It runs on a completed, approved Sanitization run, never on a raw upload.

1
Sanitization (A-01) masks a document and, via its Summarizer, emits document_metadata — a sanitized summary + free-form hints (topic/domain/tech/geo/engagement). Reviewer approves.
2
Tagging (A-02) takes that masked output + metadata hints and maps them to the controlled vocabulary, validating every tag. It is the authority on tags; Sanitization's hints are just a warm start.
3
Approved tags become the searchable index Search (A-05) and Deck (A-06) query later — by topic, never by client.

For now: manual chaining; later: Coordinator

Until the Coordinator (A-04) exists, a user triggers Tagging on a completed Sanitization run by selecting it. A-04 will chain Sanitization → Tagging automatically. The one-run-per-document contract is identical to A-01, so nothing about Tagging changes when the Coordinator arrives.

02

The contrast with Sanitization — best-match, not exhaustive

The two agents share infrastructure but have opposite failure modes, and that changes how each is built.

Sanitization (A-01)

Must be exhaustive

Every occurrence of every client identifier must be caught — a single miss is a leak. It reads every page. That's why it's token-heavy and needs the NER pre-pass + dedupe cost work.

Tagging (A-02)

Best-match is enough

Assigning "domain: retail" needs the gist, not every sentence. Tagging works off the sanitized summary + representative content, not an exhaustive scan. A missed nuance is a re-tag, not a breach — so it's cheap (one or few LLM calls) and low-risk.

Why this matters

Because Tagging is best-match, it doesn't need chunk fan-out, exact character spans, or a NER pre-pass. It largely maps Sanitization's existing metadata hints onto the validated vocabulary and fills gaps — which is why its cost is an order of magnitude below Sanitization's.

03

Input scope

  • One sanitized run per Tagging run. Input is a reference to a completed, approved Sanitization run — its masked output and document_metadata. Same one-per-run contract as A-01.
  • Never a raw document. Tagging has no fs.read_document tool and no path to the uploads directory — it can only see Sanitization's sanitized artifact. This is structural, not a policy.
  • Eligibility gate (later): the Coordinator will refuse to run Tagging on a Sanitization run that isn't completed + approved. For now the trigger endpoint enforces that check directly.
04

The controlled vocabulary — five categories, fixed and governed

A closed set of categories, each with a governed list of allowed values. Client identity is deliberately absent.

Client is not a tag — by design

Adding a client tag would reintroduce exactly the identity leak Sanitization removes and §04 of the catalogue forbids. "Organise by client" is served only through the protected client_account_id link Sanitization writes — never through the taggable, un-permissioned index Tagging produces.

05

Process — propose, hard-validate, flag, review, apply

1
Load sanitized input. The masked document + Sanitization's metadata hints. No LLM cost yet.
2
Propose tags (LLM). One pass over the sanitized summary + representative content proposes candidate tags per category with confidence, using the tagging-classification skill and the current vocabulary as the allowed menu.
3
tags.validate (protected, deterministic). Hard-rejects any tag outside the approved vocabulary — the model physically cannot apply an out-of-vocab tag. Unknown-but-plausible tags become pending_approval proposals for governance, never applied silently.
4
Flag low-confidence. Tags below threshold are flagged for reviewer confirmation, not auto-applied — "flag, don't guess," same rule as A-01.
5
Conditional review pause. If every tag is high-confidence and in-vocabulary, they auto-apply and the run completes — no pause. Only if there are low-confidence tags or new-term proposals does review.create_task file a proposal and park the run at awaiting_review for KM-reviewer / KM-governance.
6
Apply. Confirmed (or auto-applied) tags are written to run_tags and the document becomes searchable by them. audit_log.write records it.
06

Guardrails — structural, not prompted

  • Cannot emit an out-of-vocabulary tag. tags.validate is a deterministic hard gate — anything not in the approved list is rejected or downgraded to a proposal. Not a prompt instruction the model could argue around.
  • Never receives unsanitized input. Its only input path is a completed Sanitization run's masked output. No raw-document tool exists on this agent.
  • No access to masking_dictionary or source_registry. It cannot reconstruct client identity even if asked to — those calls aren't wired to it at all.
  • Low-confidence tags are flagged, never auto-applied. A reviewer confirms them before they hit the index.

The point of the validator

"Only approved tags exist in the index" is a guarantee only if the code path that writes tags rejects everything else — same reasoning as Sanitization's apply_masks being deterministic. The LLM proposes; the validator decides what can be written.

07

Note the absence

Tagging has no fs.read_document, no masking_dictionary.*, no source_registry.*. The tool surface itself is the guardrail — it reuses the shared MCP server born with A-01 but is granted only review.create_task.

09

Taxonomy governance — who owns the vocabulary

The vocabulary is a governed resource, mirroring the masking dictionary's approval flow. This maps directly onto the Step 0.5 RBAC matrix.

ActionWho
Manage tag taxonomykm_governance — add/retire/rename approved values
Propose a tagpractice_lead — proposes; lands as pending_approval
Confirm tag applicationskm_governance km_reviewer

Same feedback loop as the masking dictionary

A candidate the agent surfaces that isn't yet in the vocabulary becomes a pending_approval proposal. Governance approves it into the vocabulary; from then on the deterministic validator accepts it. The taxonomy grows by governed approval, never by the agent inventing terms at runtime.

10

Cost — an order of magnitude below Sanitization

Because Tagging is best-match and works off the summary, it's cheap. Model: Claude Sonnet 4.5 on Bedrock ($3 / 1M in, $15 / 1M out).

PassInput tokOutput tokCost
Load sanitized input + metadata0 (no LLM)0$0.00
Propose tags (summary + representative content + vocabulary menu)~8,000~800$0.036
tags.validate / apply0 (deterministic)0$0.00
Total~8,000~800≈ $0.04

With Bedrock Batch (−50%) + prompt-caching the vocabulary menu and skill (identical across every run), effectively ~$0.02 / document. No NER pre-pass, no chunk fan-out — none of A-01's cost machinery is needed here.

11

Run lifecycle

Identical state machine to Sanitization — it reuses the same worker and review infrastructure.

pending→ tagging→ awaiting_review→ applying→ completed
awaiting_review→ rejected · any state → failed
  • tagging — worker runs the propose + validate passes; no chunk fan-out, so it's fast.
  • awaiting_review — proposal filed; worker releases the run. Reviewer confirms/edits/rejects.
  • applying — on approval, tags.apply writes run_tags; cheap and inline.
12

End-to-end flow

1
User picks a completed + approved Sanitization run and submits POST /runs {agent_id:"tagging", input:{sanitization_run_id}} → pending, returns immediately.
2
Worker picks it up → tagging. Loads the sanitized output + metadata; rejects if the source run isn't approved.
3
LLM proposes tags per category against the current vocabulary; tags.validate hard-filters; low-confidence flagged.
4
review.create_task files the proposal → awaiting_review. Reviewer confirms via POST /runs/{id}/review.
5
On approve → tags.apply writes run_tags; document is now searchable by topic. Run completed. Any new-term proposals sit in the vocabulary queue for governance.

14

What A-02 shares with A-01 — why we build them together

Tagging adds a thin, cheap agent on top of infrastructure Sanitization already stands up. Building both at once means writing that shared layer once.

naviknow/
  mcp_server/ # SHARED - born with A-01; A-02 uses review.create_task only
  backend/app/
    worker.py # SHARED - DB-polling worker runs both agents
    review/ # SHARED - POST /runs/{id}/review + queue endpoints
    llm/bedrock_client.py # SHARED - Converse + prompt caching + batch
    runs/ # SHARED - lifecycle states: pending/…/awaiting_review/applying/completed
    agents/tagging/
      agent.py # orchestrates propose → validate → review → apply
      classifier.py # LLM tag proposal (single pass)
    tags/ # NEW - tag_vocabulary + validate/apply (protected calls)
    routers_tags.py # NEW - vocabulary CRUD (governance) + proposal queue

The build-together payoff

Worker, review pause, run-lifecycle states, MCP server, and the Bedrock caching/batch layer are written once and exercised by two agents. Tagging then reduces to a classifier pass + a validator + a vocabulary table — a fraction of Sanitization's surface.
